## 落地页

### 你在荣耀云空间记了那么多笔记，真的安心吗？

我用了两年荣耀云空间。784 条笔记，从商业模型到 AI 技巧，从灵感碎片到万字长文。直到有天我突然意识到一个问题：

**如果荣耀云空间停了，或者我账号出问题了，这些笔记还在吗？**

答案是：**不在**。荣耀云空间没有批量导出功能。

一条一条点、复制、粘贴？784 条，每条至少 15 秒，连续干下来要三个多小时。而且图片和音频链接全绑着 Cookie，离开浏览器就打不开了。

所以我写了这个脚本。

### 它做什么？

1. **一键开始** — 点右上角按钮，剩下的全自动
2. **自动滚屏** — 处理荣耀云空间的虚拟滚动列表，滚到底才停
3. **逐条提取** — 标题、创建时间、正文，能抓到的一篇不落
4. **图片内嵌** — 下载 → 转 Base64 → 塞进 .md，**离开浏览器也能看**；音频会保留识别文字版，虽然下不到原文件但内容不丢
5. **MIME 自动修复** — 荣耀云返回的图片类型是错的，脚本和配套工具都帮你修
6. **可暂停、可停止** — 中途随时停，已导出的全部保留
7. **文件名精确到分钟** — `待验证 AEO 技巧_6月11日 15-33.md`，不会重名

### 实测数据

| 项目      | 数据                                           |
| --------- | ---------------------------------------------- |
| 笔记总数  | 784 条                                         |
| 成功导出  | **775 条**（9 条因为内容格式异常未抓取） |
| 总耗时    | **约 3 小时**（含图片/音频下载）         |
| 导出格式  | 每篇一个 .md 文件                              |
| 图片/音频 | 全部 Base64 内嵌，离线可看                     |

### 3 小时 vs 手动 3.5 小时？那不是差不多吗？

手动 3.5 小时是你**必须全程盯着**屏幕，一条一条操作。

脚本 3 小时是**你点一下按钮就可以去干别的了**。我实测的时候就是点开始 → 下楼吃饭 → 回来收菜。

而且手动 3.5 小时你还得额外花时间把图片一张张另存为——脚本把这个也包了。

## 复盘

> 下面是我开发这个脚本过程中遇到的真实问题，和怎么一步步扒出来的。写出来不是为了显摆——是想让你知道，这个脚本是真在 784 条笔记上反复跑了无数次才成型的。

### 第一关：按钮根本不出现

我一开始以为是自己 CSS 没写好。调了半天才发现：荣耀云空间是 React SPA，URL 是 `/portal/notepad?langCode=zh-cn`，Tampermonkey 的 `@match` 对这种带查询参数的路径匹配不稳定。

后来加上 `@include` 正则才算稳了。但我发出去第一个版本之后，有人说装了还是没按钮——排查半天，是他 Tampermonkey 的"允许运行用户脚本"开关压根没开。

**踩坑总结：** SPA 页面适配，要同时考虑 `@match`、`@include`、脚本注入时机、和浏览器插件的权限层级。少了任何一环按钮都可能不出现。

### 第二关：滚动的时候进退进退、看着像卡住了

这个是虚拟滚动的锅。

左侧笔记列表是一个 React 虚拟列表，DOM 里永远只有视口可见的那十几条 `<div>`。我一开始的设计是"一次性全部滚到底再导出"——但虚拟列表会**回收 DOM 节点**，滚到底时顶部的节点已经被销毁了。

所以你能看到的现象是：滚动条时而前进、时而像在原地踏步。**这不是 bug，是虚拟列表在回收和重建 DOM 节点**——脚本在等待新节点渲染出来，一旦发现是之前处理过的标题，就继续往前推。**看起来像"停滞"，实际每一轮都在推进。**

这个来回的过程也是为什么 784 条笔记跑了 3 小时——不是脚本慢，是虚拟列表每滚一段就要重新渲染，脚本必须等它渲染完才敢点下一条，不然点到的就是重复内容。

### 第三关：同一条笔记被重复导出

虚拟列表回收 DOM 节点后，新渲染出来的 `<div>` 和上一屏的 `<div>` 是**同一个 JS 对象被复用了**。我在 DOM 上打的标记（`dataset.honorExported`）跟着旧状态一起被销毁，完全没用。

后来改成了**标题 + 创建时间的拼接字符串做唯一指纹**。点左边之前先记录右侧当前标题和时间，点完之后每 500ms 检查一次右侧内容真的变了没——如果 3 秒都没变，说明点到了已经处理过的条目，跳过。

### 第四关：图片链接离开浏览器就废了

一开始我用 `innerText` 提正文——图片全丢了。改成 `innerHTML` → Markdown 后，图片留在 .md 里了，但 URL 是远程的。

用户反馈图片不显示。原因很简单：荣耀云图片 URL 要 Cookie 鉴权，VS Code 没有这个 Cookie。

**解法：** 在页面上下文里用 `fetch` 下载图片（自动带 Cookie）→ `FileReader` 转 Base64 → 内嵌。这样 .md 是**完全自包含**的，拿到任何设备都能离线看所有图片和音频。

### 第五关：jpg 被标成 application/octet-stream

图片下载了、Base64 内嵌了，但 VS Code 预览还是不显示。

打开 .md 一看：`data:application/octet-stream;base64,/9j/4AAQ...`

`/9j/` 分明是 JPEG 魔数，但荣耀云服务器返回的 Content-Type 是 `application/octet-stream`。FileReader 原封不动沿用这个错误类型，VS Code 看到 `octet-stream` 根本不尝试渲染。

**两处修复：**

1. 脚本里加了 `fixMimeType()`，根据魔数推断正确类型后直接替换
2. 单独写了 Python 脚本并打包成 `.exe`，用来批量修复已下载 .md 里的遗留问题

### 第六关：音频下不到，但有识别文字

音频文件本身因为平台限制确实下不到——URL 是一串受保护的资源地址，`fetch` 过去拿不到原文件。

但好在荣耀云空间会在每条音频下面**自动保存一份语音识别文字**。笔记保存的时候平台已经帮你录了、识别了、存进正文了。所以虽然听不到原录音，但**核心内容基本都在**。

我在 HTML → Markdown 转换时保留了 `<audio>` 标签的占位，尽量还原音频存在的痕迹，方便你日后找回。

### 第七关：文件名冲突

一开始只用标题做文件名。结果几十条"（无标题）"全撞名了。加日期后，同一天创建的还是会撞。

后来把时间**精确到分钟**——`6月11日 15-33`。同一条笔记同一分钟创建两次的概率几乎为零，兜底的序号逻辑也一并删了。

### 写在最后

这个脚本从第一版到现在改了大大小小 20 多次。每一次改动都是因为我自己在那 784 条笔记上发现了新问题。它不完美，775/784 不是 100%——那 9 条因为内容格式异常确实没抓到——但剩下的是我能给这个场景的最实在的答案。

## 使用教程

### 第一步：装 Tampermonkey

Chrome 还没有 Tampermonkey（篡改猴）的话（ edge 也可以如此安装）：

1. 打开 [Chrome 应用商店 - Tampermonkey](https://chrome.google.com/webstore/detail/tampermonkey/dhdgffkkebhmkfjojejmpbldmpobfkfo)
2. 点「添加至 Chrome」
3. 右上角出现猴子脸图标 🐵 就装好了

### 第二步：装脚本

1. 点右上角 Tampermonkey 图标 → **管理面板**
2. 左侧 **「已安装脚本」** → 右上角 **「+」**
3. 清空默认内容，完整粘贴 `honor_notes_exporter.user.js`
4. `Ctrl + S` 保存

### 第三步：确保脚本在运行

1. 点 Tampermonkey 图标
2. 确认顶部 **「已启用」** 是蓝色/绿色
3. 确认脚本名前面有绿色勾

> ⚠️ 看不到脚本？检查 Tampermonkey 设置里的"允许运行用户脚本"开了没。

### 第四步：打开笔记页面

1. 访问 `https://cloud.honor.com/portal/notepad?langCode=zh-cn`
2. 登录荣耀账号
3. 等左侧笔记列表加载出来

### 第五步：导出

1. **右上角**出现紫色 **「📥 一键导出所有笔记」** 按钮
2. 点它
3. 脚本自动：
   - 滚左侧列表，加载所有笔记
   - 逐条点击、提取
   - 下载图片/音频 → Base64 内嵌
   - 以 `.md` 逐个保存到浏览器下载目录
4. 右上角状态条实时显示：`已导出 42 条 | 正在导出：xxx`
5. 按钮下方有 **「⏸ 暂停」** 和 **「⏹ 停止」**

> 📌 **关于滚动"停滞"和反复：** 这是正常的。左侧列表是虚拟滚动，脚本每滚一段就要等 DOM 重新渲染、验证是不是新内容。看起来像在来回踏步，实际每一轮都在推进。耐心放着就好。

### 第六步：修复 MIME（可选）

如果有些 .md 里的图片在 VS Code 不显示：

1. 把 `fix_mime_in_md.exe` 放到 .md 所在的文件夹
2. 双击运行
3. 自动扫描所有 .md，修 `application/octet-stream` → 正确 MIME

### 推荐阅读工具

| 工具               | 特点                       |
| ------------------ | -------------------------- |
| **VS Code**  | 免费，支持 Base64 图片预览 |
| **Typora**   | 所见即所得，编辑体验好     |
| **Obsidian** | 双向链接，适合搭知识库     |

## Q&A

### Q：安全吗？会不会偷我账号？

**不会。** 你可以在 Tampermonkey 编辑器里直接看全部源代码，一行都没有藏着：

- 只在你打开的 `cloud.honor.com` 上跑
- 不往任何第三方发请求
- 图片和音频只下到你本地，不传到别处
- 没加密、没混淆、没外部依赖

### Q：为什么滚动看起来在"原地踏步"？

这是虚拟列表在**回收和重建 DOM 节点**。脚本每滚一段就要等新节点渲染完、再验证是不是已经处理过的内容。看起来像停滞，但实际每一步都在往前推。**正常现象，不用管。**

### Q：为什么只导了 775 条不是 784 条？

有 9 条笔记的内容格式异常（可能是纯空白、或者编辑器状态特殊），脚本在等待右侧内容变化时超时跳过了。这是保底策略——宁愿漏几条，也不要卡死在一条上。

### Q：中途能关浏览器吗？

**不能。** 脚本需要页面开着。但你可以：

- 点 **「⏸ 暂停」** 随时停
- 切到其他标签页不影响
- 已下载的 .md 不会丢

### Q：784 条跑了 3 小时，太慢了吧？

三个小时是你**人不用管**的三个小时。点完按钮该干嘛干嘛去。而且这个速度很大程度上受限于荣耀云虚拟列表的渲染节奏——每滚一段就要等它把 DOM 重建出来——脚本已经尽可能紧凑地在点了。

如果你希望更快，可以把脚本里的 `intervalBetweenNotes` 调小（默认已经设得比较保守了，太快可能触发浏览器的下载限流）。

### Q：导出的 .md 怎么打开？

VS Code、Typora、Obsidian 都行。图片已经是 Base64 内嵌在文件里的，不联网也能看。音频保留了识别文字版，内容不丢。

### Q：为什么有些笔记叫「（无标题）」？

那是你在荣耀云空间里没写标题的笔记。脚本会用 `未命名_6月8日 14-46.md` 命名，时间精确到分钟，方便查找。

### Q：怎么更新脚本？

Tampermonkey 管理面板 → 找到脚本 → 编辑 → 全选删除 → 粘贴新版本 → 保存。

### Q：会不会被荣耀封号？

这是一个**浏览器端脚本**，和你手动一条条点、复制、粘贴做的事情一样，只是自动了。它没绕过任何安全机制，对服务器压力也不大。说白了就是个数据迁移工具。

### Q：遇到问题？

1. `F12` 打开控制台，找 `[荣耀笔记导出]` 开头的日志
2. 截图日志和页面状态
3. 带上笔记数量、浏览器版本、Tampermonkey 版本一起反馈
